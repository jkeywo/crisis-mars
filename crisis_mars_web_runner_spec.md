# Crisis: Mars Web Runner Specification

**Version:** 1.0 draft  
**Purpose:** Free-hosted web app for running *Crisis: Mars* maps, roles, action cards, resource cards, timers, and facilitator control.

---

## 1. Product Summary

Build a free-hosted web app for running **Crisis: Mars**, a 12–18 player megagame about negotiation, war, trade routes, corporate interests, Martian independence, and asteroid-belt politics.

The app should not replace live negotiation. It should replace or supplement the paper control layer: map score tracking, turn timing, action-card placement, action queues, resource-card lending/trading, role/brief access, facilitator prompts, opportunities, news, end-of-turn updates, and correction tools.

The app is intended for **in-room play**. Players should still move around and negotiate face-to-face. The app should not include private chat or phone-based player communication.

---

## 2. Core Design Principles

### 2.1 Hybrid physical/digital play

The website should support the printed megagame experience rather than replacing it entirely.

Physical components should remain useful:

- Printed role name badges.
- Printed resource cards.
- Printed or projected maps.
- Team areas.
- Live facilitator presence.
- In-person negotiation.

The app handles:

- Digital role and faction brief access.
- Digital map score state.
- Digital action-card placement.
- Action queues and player call alerts.
- Resource-card ownership ledger.
- Resource lending, recall, reclaim, and spending.
- NPC resource pools.
- QR-based trade/loan/reclaim flows.
- Timer and phase control.
- Facilitator news/upkeep prompts.
- Full card correction and recovery tools.
- Exportable game log.

### 2.2 QR-first interaction

QR codes should be the default for:

- Joining a game.
- Unlocking a role.
- Rejoining from another device.
- Identifying physical cards.
- Initiating trades or loans.

NFC may be added later as an optional shortcut, but must not be required for core play.

### 2.3 No player messaging

The app must not contain player-to-player chat, faction chat, direct messages, or secret mobile communication.

Allowed communication features:

- System alerts.
- Facilitator announcements.
- Trade/loan prompts generated by physical proximity.
- Action call-up alerts.
- Phase warnings.
- Resource recall alerts.

## 3. Recommended Technical Architecture

### 3.1 Frontend

Recommended:

- Static Progressive Web App.
- React/Vite, SvelteKit static adapter, Vue, or equivalent.
- Hosted on Cloudflare Pages, Netlify, Vercel, or GitHub Pages.
- Service worker for offline caching.
- Web Push optional.

### 3.2 Backend

Recommended backend: **Supabase**.

Use:

- Postgres for relational game state.
- Supabase Realtime for map/timer/card updates.
- Supabase Storage for maps, role images, card images, and scenario assets.
- Row-Level Security for private briefs, role access, and facilitator-only data.
- Anonymous/player-token access rather than full player accounts.

Alternative backend: **Firebase**.

Use Firebase if the developer prefers:

- Firestore realtime sync.
- Firebase Hosting.
- Anonymous Firebase Auth.
- Firebase Cloud Messaging.

Supabase is the cleaner fit for this game because the data is strongly relational: factions, roles, cards, card ownership, map scores, opportunity triggers, action submissions, and audit events.

### 3.3 Server-authoritative state

The server must be the source of truth for:

- Player role assignment.
- Current hand.
- Loaned-out resources.
- Borrowed resources.
- Spent resources.
- Reclaim rights.
- Action-card placement.
- Current map scores.
- Timer/phase state.
- NPC resources.
- Audit log.

Browser local storage should only cache data for convenience. It must never be the only record of a player’s state.

## 4. Game Structure

### 4.1 Runtime schedule

The app should ship with the built-in *Crisis: Mars v1.0* schedule:

| Segment | Time |
|---|---:|
| Prologue / briefing / casting | 30 minutes |
| Turn 1 | 20 minutes |
| Turn 2 | 20 minutes |
| Turn 3 | 20 minutes |
| Turn 4 | 20 minutes |
| Epilogue / debrief | 10 minutes |

Each turn contains:

| Phase | Duration | Main App Function |
|---|---:|---|
| Team Phase | 5 minutes | News, team planning, opportunities |
| Negotiation Phase | 5 minutes | Trading, resource reclaiming, action-card placement |
| Action Phase | 10 minutes | Action queue, 60-second action calls, resolution |

The facilitator dashboard must allow:

- Start.
- Pause.
- Resume.
- Skip phase.
- Add time.
- Remove time.
- Jump to phase.
- End game.
- Export game state.

### 4.2 Facilitator roles

The app should support 2 or 3 facilitators.

#### Main Facilitator / Earth Control

Responsible for:

- Overall timing.
- Earth map.
- Canopy Corp.
- UNSS.
- Earth and war news.
- UN Ambassador NPC, if needed.

#### Mars Control

Responsible for:

- Mars map.
- Viva Mars.
- Deimos Rising Corp.
- Mars news.
- Martian Senate Speaker NPC, if needed.

#### Belt Control

Responsible for:

- Asteroid Belt map.
- Belt Union.
- Free Federation.
- Belt news.
- Shipping/trade route adjudication.

If only two facilitators are present, the app should allow Earth Control and Mars Control to share Belt responsibilities.

## 6. Role Name Badges and QR Unlocking

### 6.1 Physical badge structure

Each player should receive a physical **role name badge** during casting.

The badge front should show only public information:

- Faction name.
- Faction icon/colour.
- Role name.
- Role code, e.g. C1, U2, V3.
- Optional player name field.

The badge back should contain:

- A QR code.
- A short manual unlock/recovery code.
- A warning: **Private. Do not show this side to other players.**

The badge QR unlocks that specific faction and role in the app.

### 6.2 QR token principle

The role badge QR should contain a token, not private text.

It should not contain:

- The private brief.
- Card state.
- Score data.
- Hidden facilitator information.

It should contain or resolve to:

- Game/session token.
- Role access token.

Example URL shape:

```text
https://crisis-mars.app/join/g/{gameToken}/r/{roleToken}
```

### 6.3 Badge-first casting flow

Recommended live casting flow:

1. Facilitator lays out faction areas with role badges.
2. Players choose factions physically.
3. Teams decide among themselves who takes which role.
4. Player takes the corresponding name badge.
5. Player scans the QR on the back.
6. App shows: “You are claiming: [Faction] — [Role].”
7. Player enters display name.
8. Player confirms.
9. App unlocks that player’s private faction and role material.
10. Facilitator dashboard marks the role as claimed.

Players should not be able to browse unclaimed private briefs inside the app.

### 6.4 Claim safeguards

The app should support:

- One active role claimant per role.
- Facilitator reset if the wrong player scans a badge.
- Manual unlock code entry if QR scanning fails.
- Facilitator reissue if a phone is lost or refreshed.
- Facilitator lock if a badge is lost.
- Facilitator token rotation.
- Audit events when roles are claimed, reset, locked, rotated, or transferred.

## 7. QR Rejoin and Device Recovery

### 7.1 Requirement

A player scanning the same role badge QR code should restore their full game state, even from a different device.

The role QR is therefore both:

- A role claim token.
- A role recovery token.

### 7.2 Restored state

Scanning the same badge QR should restore:

- Player identity for that role.
- Faction assignment.
- Role assignment.
- Private faction and role brief access.
- Current hand.
- Borrowed cards.
- Loaned-out cards.
- Spent reclaimable cards.
- Action-card state.
- Current map placement.
- Current alerts.
- Opportunity access, if relevant.
- Reclaim-used status for the current turn.
- Push notification setup prompt for the new device.

### 7.3 First scan

1. Player scans badge QR.
2. App opens role claim page.
3. App shows faction and role name.
4. Player enters display name.
5. Player confirms claim.
6. Server creates or binds a participant record.
7. App unlocks role and current game state.

### 7.4 Later scan on same device

1. Player scans badge QR again.
2. App recognises the role.
3. App opens directly into the player dashboard.
4. No state is reset.

### 7.5 Later scan on another device

1. Player scans badge QR.
2. App recognises that the role is already claimed.
3. App shows: “Restore access to [Faction] — [Role]?”
4. Player confirms.
5. App restores the same participant state from the server.
6. Previous device remains valid unless single-device mode is enabled.
7. App prompts the player to enable alerts on the new device.

### 7.6 Misuse controls

Recommended default:

- Allow multiple devices for the same role.
- Alert facilitators if the same role is active on more than one device.
- Let facilitators revoke suspicious sessions.

Optional stricter mode:

- Single active device per role.
- Scanning the QR on a new device logs out the old device.

## 9. Player Resource Screen

### 9.1 Purpose

The player resource screen should not simply show the cards currently in the player’s hand. It should show every resource card the player has a meaningful claim over.

The screen should distinguish:

- Possession.
- Ownership.
- Borrowing.
- Recall rights.
- Reclaim rights.
- Spent state.

### 9.2 Required sections

The screen should be divided into four sections:

1. **In My Hand**
2. **Borrowed From Others**
3. **Loaned Out — Can Recall**
4. **Spent — Can Reclaim**

Each section should collapse/expand on mobile.

During Negotiation Phase, move **Spent — Can Reclaim** near the top with a clear prompt:

> You may reclaim 1 of your spent resource cards this phase.

During Action Phase, move **In My Hand** near the top because players need to commit resources quickly.

### 9.3 In My Hand

Shows all resource cards currently held by the player.

Includes:

- Cards originally owned by the player.
- Cards loaned to the player.
- NPC resources currently held by the player.
- Any resources received through opportunities or facilitator actions.

Each card should show:

- Card name.
- Faction colour/icon.
- Original owner.
- Current holder.
- State.
- Whether it can be spent.
- Whether it can be traded or loaned.
- Whether the original owner can recall it.

Available actions:

- Offer / trade / loan.
- Commit to action.
- Scan card QR.
- Show transfer QR.

### 9.4 Borrowed From Others

Shows cards currently held by the player but originally owned by someone else.

Each card should clearly warn:

> This card can be recalled by its original owner until spent.

Available actions:

- Use in action.
- Return voluntarily.
- Offer in trade only if rules permit.
- View original owner.

Default rule: borrowed cards should not be re-loaned unless the facilitator enables it.

### 9.5 Loaned Out — Can Recall

Shows resource cards originally owned by the player but currently held by another player, team, or NPC.

Each card should show:

- Card name.
- Current holder.
- Current holder’s faction/role.
- Whether it is still unspent.
- Whether it is currently committed to an unresolved action.
- Recall status.

Available actions:

- Recall now.
- Voluntary gift/permanent transfer, if facilitator permits.
- View audit history.

Recall rules:

- A player may recall their own loaned resource card at any time until it is spent.
- If the card is committed to an unresolved action, recall should either be blocked or require facilitator approval.
- If the card has already been accepted and spent by a facilitator, it moves into **Spent — Can Reclaim**.

Recall flow:

1. Player presses “Recall”.
2. App checks the card is originally owned by the player.
3. App checks the card is not spent.
4. App checks the card is not locked into a resolved action.
5. Card returns to owner’s hand.
6. Current holder receives an in-app alert.
7. Audit log records the recall.

### 9.6 Spent — Can Reclaim

Shows resource cards originally owned by the player that are currently discarded/spent.

This section should be visible at all times, but the reclaim button should only be active during the Negotiation Phase.

Each card should show:

- Card name.
- Turn spent.
- Action it was spent on.
- Facilitator who accepted it.
- Whether it is eligible for reclaim.
- Whether the player has already reclaimed a card this turn.

Available actions:

- Reclaim during Negotiation Phase.
- View action history.
- Filter by faction/type.

Reclaim rules:

- A player may reclaim one of their own discarded resource cards during the Negotiation Phase.
- The app should track whether the player has already used their reclaim this turn.
- Reclaimed cards move directly into **In My Hand**.
- Cards discarded by another player cannot be reclaimed unless the reclaiming player is the original owner.

Reclaim flow:

1. Negotiation Phase begins.
2. App marks eligible spent cards as reclaimable.
3. Player chooses one card.
4. App confirms: “Reclaim this resource for Turn X?”
5. Card returns to hand.
6. Player’s reclaim for the turn is marked as used.
7. Audit log records the reclaim.

### 9.7 Resource screen state logic

A card appears in **In My Hand** if:

- `current_holder_participant_id = current player`
- `current_state = held`

A card appears in **Borrowed From Others** if:

- `current_holder_participant_id = current player`
- `original_owner_participant_id != current player`
- `current_state = held`

A card appears in **Loaned Out — Can Recall** if:

- `original_owner_participant_id = current player`
- `current_holder_participant_id != current player`
- `current_state = held`
- `is_recallable = true`

A card appears in **Spent — Can Reclaim** if:

- `original_owner_participant_id = current player`
- `current_state = spent`
- `reclaimed_turn is null`

The reclaim button is enabled only if:

- Current phase is Negotiation Phase.
- Card is in `spent` state.
- Original owner is the current player.
- Player has not reclaimed a resource this turn.
- Facilitator has not locked reclaiming.

## 10. Cards

### 10.1 Card types

The app needs two major card types:

1. Action Cards
2. Resource Cards

### 10.2 Action cards

Each player has one role-specific action card.

Action-card rules:

- It starts in the player’s possession.
- During the Negotiation Phase, the player must place it on one of the maps.
- During the Action Phase, the player is called to the map where the action card was placed.
- After the action resolves, the action card returns to the player.
- A player should not be able to place more than one action card per turn.
- A player should be flagged if they have not placed their action card by the end of Negotiation Phase.

Digital action-card states:

- In hand.
- Placed: Earth.
- Placed: Mars.
- Placed: Belt.
- Called.
- Resolving.
- Resolved.
- Returned.

### 10.3 Resource cards

Each player starts with 5 resource cards.

Resource-card rules:

- A resource card can be loaned/traded to another player at any time.
- A player can recall their own loaned resource card at any time until it is spent.
- A used resource card is discarded.
- During the Negotiation Phase, a player may reclaim one of their own discarded resource cards.
- There is no hand limit.
- Resource cards usually add +1 impact if accepted by the facilitator.
- Resource cards should be from different factions when used on the same action.
- Facilitators can reject irrelevant resources, returning them unused.

Digital resource-card states:

- Owned by original role.
- Held by player.
- Loaned to player.
- Offered in trade.
- Committed to action.
- Accepted by facilitator.
- Rejected by facilitator.
- Discarded/spent.
- Reclaimed.
- Removed/voided.

## 11. Resource QR and Trading

### 11.1 Card QR principle

Role-badge QR codes and resource-card QR codes are separate token types.

- **Role QR:** unlocks role access and recovery.
- **Resource QR:** identifies and moves an individual card.

A resource QR should contain only an opaque card token.

### 11.2 Scanning a resource card

Scanning a resource card should show:

- Card name.
- Faction colour/icon.
- Original owner role.
- Current holder.
- Current state.
- Eligible actions based on permissions.

Possible actions:

- Loan.
- Trade.
- Return.
- Recall.
- Mark spent, facilitator only.
- Attach to action.
- View audit history.

### 11.3 Loaning a resource

1. Player A scans their own card or opens it in their hand.
2. Player A selects “Loan to another player”.
3. Player B scans the generated QR code or confirms from a local trade screen.
4. Player B confirms receipt.
5. Card appears in Player B’s hand as borrowed.
6. Player A sees it under **Loaned Out — Can Recall**.

### 11.4 Recalling a resource

1. Original owner opens **Loaned Out — Can Recall**.
2. Selects a loaned-out card.
3. Presses “Recall”.
4. App validates the card is not spent.
5. Card returns to original owner’s hand.
6. Previous holder receives an alert.
7. Audit log records the recall.

### 11.5 Spending a resource

1. During action resolution, facilitator sees committed resources.
2. Facilitator accepts or rejects each resource.
3. Accepted resources move to spent/discarded.
4. Rejected resources return to current holder.
5. Spent resources become eligible for one-per-turn reclaim by their original owner during Negotiation Phase.

### 11.6 Player-to-player trade

1. Player A selects “Trade”.
2. Player A chooses cards to offer.
3. App generates a temporary trade QR code.
4. Player B scans the QR code.
5. Player B chooses cards to offer in return.
6. Both players confirm.
7. App validates ownership and card state.
8. Cards transfer simultaneously.
9. Both players receive confirmation.

Trade QR links should expire quickly.

### 11.7 NPC-player trade

NPC trades are confirmed by the controlling facilitator.

1. Facilitator opens NPC resource panel.
2. Facilitator selects NPC cards to offer.
3. Player scans QR or opens local trade prompt.
4. Player selects offered cards, if any.
5. Facilitator confirms.
6. App transfers cards.
7. Audit log records NPC deal.

## 12. NPC Resource Pools

### 12.1 NPCs as pseudo-players

The app should treat the two facilitator NPCs as pseudo-players with full resource-card inventories.

NPCs:

- N1: UN Ambassador
- N2: Martian Senate Speaker

Each NPC should have:

- Role card.
- Public description.
- Private facilitator notes.
- Resource-card hand.
- Trade/loan/reclaim controls.
- Audit log.
- Map/opportunity associations.
- Controlling facilitator.

Default controller:

- UN Ambassador: Main Facilitator / Earth Control.
- Martian Senate Speaker: Mars Control.

### 12.2 NPC resource rules

NPC resources should be fully tradeable unless a specific scenario rule marks them otherwise.

NPCs may:

- Loan resources to players.
- Trade resources with players.
- Receive resources from players.
- Spend resources on behalf of NPC-backed deals.
- Reclaim resources they originally own, if they were loaned.
- Have resources discarded through action consequences.
- Receive resources through opportunities, tithes, or facilitator fiat.

NPC resources should behave like player resources in the ledger:

- Original owner: UN Ambassador or Martian Senate Speaker.
- Current holder: player, faction, or NPC.
- State: held / loaned / committed / spent / discarded / reclaimed.
- Visibility: facilitator-visible, holder-visible, and optionally public if facilitator marks the deal public.

### 12.3 NPC resource panel

Facilitator dashboard should include an **NPC Resources** panel.

For each NPC, show:

- Current hand.
- Loaned-out cards.
- Loaned-in cards.
- Discarded cards.
- Pending trades.
- Active promises/deals.
- Reclaim buttons.
- Create QR trade button.
- Transfer to player button.
- Mark spent button.

   - Mars Trade Route below 4: War Progress cannot end lower than it started.
5. Facilitator confirms, edits, or cancels the change.
6. App records the update in the audit log.

## 14. Turn and Phase Behaviour

### 14.1 Team Phase

The app should:

- Move all players into Team Phase.
- Display “stay with your team and plan”.
- Trigger facilitator news prompts for each map.
- Show generated opportunities to facilitators.
- Let facilitators reveal opportunities to the relevant faction/team.
- Track unresolved opportunity choices.
- Apply scripted war correspondence prompts.

#### War correspondence prompts

Turn 1:

- Martian Senate declares independence.
- Earth Gov denounces the declaration as illegal.

Turn 2:

- UN Congress declares war on Mars.
- Start War Progress on the Mars map at 11.
- Add +6 Earth Gov Military.
- Apply -4 UN Shipping Control at the Asteroid Belt.

Turn 3:

- Martian Senate reveals secret fleet construction on Phobos.
- Add +5 Senate Military.
- Facilitator may delay or cancel this if needed for balance.

Turn 4:

- Do not mechanically meddle in the war.
- Announce how UN and Senate are reacting.
- Remind players this is the final turn.

These should be editable facilitator prompts, not mandatory automatic changes.

### 14.2 Negotiation Phase

The app should:

- Show all players the current phase and remaining time.
- Allow players to trade/loan resource cards.
- Allow each player to reclaim one discarded resource card they originally own.
- Allow each player to place their action card on Earth, Mars, or Belt.
- Show players whether their action card has been placed.
- Show facilitators a live list of unplaced action cards.
- Issue warnings at 2 minutes and 1 minute remaining.
- Lock action-card placement when the phase ends, unless a facilitator unlocks it.

No player-to-player messaging should exist.

### 14.3 Action Phase

The app should:

- Build an action queue from placed action cards.
- Allow facilitators to randomise by map or globally.
- Support three map queues: Earth, Mars, Belt.
- Call players to the appropriate facilitator/map.
- Alert the called player.
- Start a 60-second action timer.
- Let the facilitator record:
  - Stated fictional action.
  - Allies involved.
  - Teams/factions represented.
  - Resources proposed.
  - Resources accepted/rejected.
  - Difficulty modifier.
  - Consequence die result.
  - Final impact.
  - Result tier.
  - Score changes.
  - Resource discards.
  - Fallout notes.
  - News flag.
  - Future modifier flag.

After resolution, the player’s action card returns to hand.

## 15. Action Resolution Assistant

The app should not fully automate adjudication. It should assist facilitators.

### 15.1 Default impact formula

Default formula:

```text
Current Turn Number + 1 per Ally + 1 per Accepted Resource - Difficulty = Final Impact
```

The formula should be configurable because different game sheets may phrase impact slightly differently.

### 15.2 Resource validation warnings

The resolution assistant should warn if:

- Two resources from the same faction are proposed.
- More than 3 resources are proposed.
- A resource is not held by the acting player or ally.
- A resource is already spent.
- A resource belongs to another player and has been recalled.

The facilitator can override all warnings.

### 15.3 Consequence die

The app should support three outcome types:

- Unexpected consequence.
- Normal resolution.
- Increase impact.

The facilitator should be able to select the die result manually or press “roll digitally”.

### 15.4 Result tiers

Default simple result tiers:

| Final Impact | Result |
|---:|---|
| 1 or less | Insignificant |
| 2–3 | Minor |
| 4–5 | Moderate |
| 6–7 | Notable |
| 8–9 | Major |
| 10+ | Radical |

The app should display a result description box for the facilitator to write the actual fictional outcome.

### 15.5 Action-effect templates

The facilitator can select one or more effect templates.

#### Modify map scores

Suggested score movement by result tier:

| Result | Score Modifier |
|---|---:|
| Insignificant | +/-1 |
| Minor | +/-2 |
| Moderate | +/-3 |
| Notable | +/-4 |
| Major | +/-5 |
| Radical | +/-6 |

#### Regain resources

Restore narratively appropriate resources.

#### Sabotage resources

Mark another player’s or faction’s resource card as discarded. Use sparingly.

#### Prepare for the future

Create a future action modifier attached to:

- Player.
- Faction.
- Map.
- Location.
- Target score.
- Future action type.

#### Make the news

Flag the action for the next Team Phase news prompt.

#### Change upkeep rules

Rare effect. Requires facilitator confirmation. Can alter opportunity thresholds, end-of-turn rules, or trade-route behaviour.

## 17. Belt Union Tithe

The Belt Union tithe should be implemented as a structured NPC resource transfer.

Default:

- Turns 1–2: Belt Union owes 1 resource card per turn to the UN Ambassador.
- Turns 3–4: increase to 2 resource cards by default, with facilitator override.

App flow:

1. During the relevant phase, the app prompts Earth Control: “Request Belt Union tithe.”
2. Belt Union selects the required resource card or refuses.
3. If accepted, the resource transfers to the UN Ambassador’s hand.
4. If refused, the app creates a retaliation/fallout prompt.
5. Audit log records the outcome.

This makes the UN Ambassador’s resource pool materially important and gives the tithe a visible destination.

## 28. Implementation Notes

### 28.1 Token separation

The system must keep these tokens distinct:

| Token Type | Purpose |
|---|---|
| Game join token | Lets players reach the session |
| Facilitator token | Grants facilitator dashboard access |
| Role badge token | Unlocks/rejoins a specific role |
| Manual role code | Fallback for badge QR failure |
| Resource card token | Identifies a physical card |
| Trade token | Temporary QR for a proposed trade |

### 28.2 Ownership versus holding

The app must distinguish:

- **Original owner:** who can recall/reclaim the card.
- **Current holder:** who physically/digitally has the card now.
- **Controller:** who can act on behalf of an NPC or facilitator-owned card.

This distinction is central to Crisis: Mars because players can loan resources, recall them, spend them, and later reclaim them.

### 28.3 Facilitator authority

Facilitators must be able to override any card or score state, but all overrides must be audited.

The app should make corrections easy, but never invisible.

### 28.4 Free-hosting constraint

The app should minimise backend load:

- Store compact game state.
- Store large images in object storage.
- Cache rules/maps on client.
- Avoid unnecessary high-frequency writes.
- Use realtime only for live state that matters.
- Archive or export completed games.

## 29. Open Questions

1. Should physical resource cards each have their own QR code, or should QR codes be printed only on role sheets and managed digitally?
2. Should borrowed cards be re-loanable, or should only the current holder and original owner interact with them?
3. Should action cards be physical, digital, or both?
4. Should map score changes be visible instantly to all players, or only after facilitator news updates?
5. Should players be able to propose action text in-app before being called, or should all action explanation remain live at the map?
6. Should NPC resources start with fixed decks or be created by facilitator judgement?
7. Should the UN Ambassador and Martian Senate Speaker have role badge QR codes for facilitator convenience?
8. Should a facilitator be able to impersonate a player view for troubleshooting?
9. Should the app print badges/cards directly from scenario data?
10. Should finished games be saved in the app, or exported and deleted by default?
